package.json 版本号说明

关于版本号的描述

npm 模块的完整版本号一般是【主版本.次要版本.补丁版本】，一般情况下，次要版本号发生改变的话，表示程序有重大更新。

语义版本号

使用NPM下载和发布代码时都会接触到版本号。NPM使用语义版本号来管理代码，这里简单介绍一下。
语义版本号分为X.Y.Z三位，分别代表主版本号、次版本号和补丁版本号。当代码变更时，版本号按以下原则更新。

  1. 如果只是修复bug，需要更新Z位。
  2. 如果是新增了功能，但是向下兼容，需要更新Y位。
  3. 如果有大变动，向下不兼容，需要更新X位。
    版本号有个这个保证后，在申明第三方包依赖时，除了可依赖于一个固定版本号外，还可以依赖于某个范围的版本号。例如：”argv”:”0.0.x”表示依赖于0.0.x系列的最新版argv。

使用～表示版本范围

不改变大版本和次要版本号

这里大概可以如此概述： 1⃣️ 补丁版本号缺失，则允许补丁版本号升级； 2⃣️ 次要版本号+补丁版本号缺失，则允许次要版本号+补丁版本号升级；
～2.3.4
  • 描述：主版本+次要版本+补丁版本 （2.3.4 <= version < 2.4.0）
  • 说明：在主版本+次要版本不允许变更的前提下，允许补丁版本升级（补丁版本号下限是4，无上限）。
～2.3
  • 描述：主版本+次要版本 （2.3.0 <= version < 2.4.0）
  • 说明：在主版本+次要版本不允许变更的前提下，允许补丁版本升级。
～2
  • 描述：主版本 （2.0.0 <= version < 3.0.0）
  • 说明：在主版本不允许变更的前提下，允许次要版本+补丁版本升级。

使用^表示版本范围

不改变大版本号

1⃣️ 若主版本号不为0，则允许次要版本号+补丁版本号升级； 2⃣️ 若主版本号为0，次要版本号不为0，则允许补丁版本号升级； 3⃣️ 若主版本号+次要版本号皆为0，将无法升级模块； 4⃣️ 若主版本号不为0，补丁版本号缺失（将被视为0），那么将允许次要版本+补丁版本升级到最新； 5⃣️ 若主版本为0，补丁版本缺失（将被视为0），那么允许补丁版本升级到最新； 6⃣️ 若次要版本+补丁版本均缺失，此时补丁版本被视为1，那么将允许次要版本+补丁版本升级到最新。
^1.3.4
  • 描述：主版本号不为0 （1.3.4 <= version < 2.0.0）
  • 说明：主版本不为0，允许次要版本+补丁版本升级
^0.2.3
  • 描述：主版本号为0，次要版本号不为0 （0.2.3 <= version < 0.3.0）
  • 说明：主版本为0，次要版本不为0，允许补丁版本升级
^0.0.3
  • 描述：主版本号+次要版本号均为0 （0.0.3 <= version < 0.0.4）
  • 说明：主版本号+次要版本号均为0，无法升级模块
^1.3
  • 描述：主版本号不为0，补丁版本号缺失 （1.3.0 <= version < 2.0.0）
  • 说明：主版本号不为0，补丁版本因缺失被视作0，允许次要版本+补丁版本升级到最新
^0.2
  • 描述：主版本为0，补丁版本缺失 （0.2.0 <= version < 0.3.0）
  • 说明：主版本为0，补丁版本因缺失被视作0，允许补丁版本升级到最新
^1
  • 描述：主版本号不为0，次要版本+补丁版本均缺失 （1.0.0 <= version < 2.0.0）
  • 说明：主版本不为0，次要版本+补丁版本因缺失被视作0，允许次要版本+补丁版本升级
^0
  • 描述：主版本为0，次要版本+补丁版本均缺失 （0.0.1 <= version < 1.0.0）
  • 说明：主版本为0，次要版本因缺失被视作0，补丁版本虽缺失但是智能被视作1，允许缺失的次要版本+补丁版本升级到最新
